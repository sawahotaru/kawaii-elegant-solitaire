## 🔍 重大な不具合の修正記録 (デバッグ・フェーズ)

実機テスト中に発見された複数の複雑なバグに対する修正プログラムです。

### バグA: カードの操作ロック（一度動かすと反応しなくなる）
- **現象**: カードを他の列に移動させた直後、そのカードや新しく表になったカードがドラッグできなくなる。
- **原因**: Reactのコンポーネント再利用（Reconciliation）により、`useDraggable` の内部状態が古い場所の情報を保持し続けていた。
- **対策**: `key` 属性と `draggableId` をユニーク化。さらに、カードが表裏反転した瞬間にコンポーネントを強制再生成する「Flip-Stable Key」を導入し、完全なリセットを保証。

### バグB: カードの消失と移動スナップバック
- **現象**: 素早い操作をするとカードが消える、あるいは元の場所に戻される。
- **原因**: 状態更新時のシャローコピーによるオブジェクト参照の重複が、`dnd-kit` の追跡を混乱させていた。
- **対策**: ステートエンジン（`useGameState`）におけるすべての更新を **Deep Clone (深いコピー)** に刷新。データの独立性を確保。

### バグC: 特定の列がクリックできない（右から2列目）
- **現象**: 画面サイズによって、右から2列目のカードが掴めない。
- **原因**: 右隣の列の透明な余白が、重なり順の影響で左側の列を遮っていた。
- **対策**: Tableau列の `z-index` を反転（左側を高く設定）。左側の列が常に手前に来るようにし、クリックの遮断を解消。

### バグD: ドラッグ終了の失敗
- **現象**: ドラッグを離してもカードが指に付いたまま、あるいは不安定になる。
- **原因**: ドラッグ中に表示されるオーバーレイ要素がマウスイベントを遮断していた。
- **対策**: `DragOverlay` に `pointer-events-none` を適用し、背景のドロップ判定を確実に受け取れるように調整。

## 🔬 追加の検証と修正（解析フェーズ）

実コードによるロジック実行テスト（Node）と、ブラウザ自動化（Playwright）による実機テストを通じて、追加で2件の不具合を発見・修正しました。

### バグE: ヒントが実行不可能な手を提示する
- **現象**: `findValidMoves` が、上に他のカードが乗って「埋もれている」表向きカードまで組札への移動候補に含めていた。
- **影響**: 実際には単独で動かせないカードをヒントとして提示し得る。
- **対策**: 組札への候補を「各列の一番上のカード」＋「waste のトップ」に限定。場札への候補（連番スタックの先頭になれる表向きカード）は従来通り維持。Node のロジックテスト 41/41 で回帰確認。

### バグF: 山札クリックで2枚ドローされる（二重発火）
- **現象**: Stock を1回クリックすると2枚ドローされ、Moves が +2 されていた（上級モードでは1クリック6枚）。
- **原因**: Stock 領域の外側 `div` と、その上に重なる内側 `Card` の**両方に `onClick={drawCards}`** が設定されており、クリックイベントがバブリングして 2 回発火していた。
- **対策**: 内側 `Card` の重複した `onClick` を削除（外側 `div` が領域全体のクリックを処理）。ブラウザ実機で「1クリック=1ドロー」を確認。

### 公開（OSS化）対応
- `LICENSE`（MIT）、`.gitignore`、GitHub Actions による GitHub Pages 自動デプロイ（`.github/workflows/deploy.yml`）を追加。
- `vite.config.ts` に `base: './'` を設定し、GitHub Pages 等のサブパス配信に対応。
- `package.json` に公開用メタデータ（description / license / keywords / repository 等）を追加。
- README を実装に合わせて更新（コンフェッティは Framer Motion 実装である旨を反映、ライブデモ／ライセンス／導入手順を追記）。

## 🆕 2026-06-28 大型アップデート（難易度リデザイン・アシスト機能・スマホ対応）

「ランダム配布で詰みが多くクリアしづらい」というフィードバックを起点に、難易度設計を見直し、複数のアシスト機能を追加。検証は **React fiber から内部状態（伏せ札・山札順を含む全カード）を抽出**し、自作の可解判定ソルバーと突き合わせる手法で、ブラウザ実機（Playwright）にて定量的に実施した。

### 設計判断: 「解ける保証 × やり直せる(Undo)」
カジュアル体験で「理不尽」を避ける本質は、難易度の数値ではなく **解ける盤面の保証** と **取り返せる Undo** の2軸であると整理。これを初心者・中級の土台に採用し、純ランダム（解けない局あり）の理不尽は上級にオプトインで隔離した。

### バグG: 山札リサイクル順の逆転
- **現象**: 山札を引き切ってリサイクルすると、直前に見えていた最後の捨て札が即再表示され、循環順が標準と逆だった。
- **原因**: `useGameState.ts` の `currentWaste.reverse()`。ドローは先頭から・捨て札は末尾追加のため、リサイクルで reverse すると順序が逆転していた。
- **対策**: `.reverse()` を除去し、最初にめくった札から再ドローされる標準順に修正（カードの消失・重複なしを実機で確認）。

### バグH: ヒントが進展しない手を提示
- **現象**: ヒントが「伏せ札をめくらない空列への K 移動」など無意味な合法手を提示することがあった。
- **原因**: `findValidMoves` の先頭手を無条件採用していた。
- **対策**: 手に優先度を導入（組札＞めくり/手札出し＞載せ替え＞空列シャッフル）。常に最良クラスの手を提示するよう変更。多数の局面で「最小優先度の手＝ヒント」を確認。

### 機能: 解ける盤面ジェネレータ（`src/utils/solver.ts` 新規）
- 最良優先探索（ノード予算上限つき）の Klondike ソルバーを実装。**合法手のみ生成するため「solvable=実在の勝ち筋あり」を保証**。
- `initializeGame` を `generateDeal` 経由に変更し、初心者・中級は「解けた盤面のみ」を採用（上級は判定なし）。初心者は小さい探索予算で「やさしい盤面」に寄せている。
- 実機検証で初心者・中級の生成盤面が独立ソルバーで 100% 可解、生成時間も実用域（中級 中央値 ~数十ms）を確認。

### 機能: 難易度選択UI / Undo 緩和
- ヘッダーに難易度（初心者 / 中級 / 上級）セレクタを追加し、`localStorage` に保存（既定は初心者）。従来は `normal` 固定で UI から切替不可だった。
- 中級の Undo 上限を 10 → 100 に緩和（ミスから取り返しやすく）。

### 機能: 一括あがり（オートコンプリート）
- すべての札が表向きになると「一括あがり」ボタンを表示。`solveBoards` が勝ち筋の盤面列を求め、盤面を順次差し替えてアニメーション再生し勝利演出まで到達。
- 勝ち筋が全 52 枚完成に到達することを独立再現で確認。勝利演出（Perfect! / Play Again / 紙吹雪 / ファンファーレ）もモバイル実機で目視確認。

### 機能: ヒント上限の設定
- ヘッダーで `自動 / 3 / 5 / 10 / ∞` を選択可能（全難易度に適用・保存）。`自動` は難易度準拠。変更時は現ゲームの残数も即反映。ヒント残数バッジは有限のとき常時表示。

### 機能: 初心者限定「カード入れ替え（ズル）」
- 🪄ボタン → 盤面の表向きカードをタップ → ピッカーで欲しいカードを選ぶと2枚を入れ替え。
- 重複が出ないよう正体（suit/rank/id）を交換し、各スロットの表裏は維持。**組札のカードは対象外**で完成列を保護。**初心者のみ**（UI・ロジック両方でガード）、**Undo で復元可**。

### スマホ表示の修正
- タブローの重なり量が固定px（PCのカード高前提）でモバイルでは表向きカードが完全に重なり読めなかった問題を、`matchMedia` による画面幅連動の重なり量に変更して解消。
- モバイルのカード幅を列幅に合わせて縮小（はみ出し・隣列との重なりを解消）。PC表示は据え置きを計測確認。

### 検証手法のメモ
- ブラウザ自動化（Playwright）で `__reactFiber$` から `useGameState` の state を抽出し、伏せ札・山札順を含む全カードを取得して可解性・整合性（52枚ユニーク等）を定量チェック。
- 一様シャッフルの妥当性も実測（赤割合・単色列頻度が理論値と一致）。難易度操作は無し。
